# Обновление системы оценки RAG (22.04.2026)

Сегодня был проведен комплекс работ по стабилизации и расширению возможностей пайплайна оценки качества ответов.

## Основные доработки:

### 1. Исправление потери данных (Data Flow Fix)
- **Файл**: `eval/core/api_utils.py`
- **Проблема**: При онлайн-прогонах скрипт сохранял ответы бота, но не переносил поле `expected_output` (эталонный ответ) в итоговый лог. Это приводило к невозможности расчета метрик Contextual Precision и Contextual Recall в оффлайн-режиме.
- **Решение**: Добавлен проброс полей `expected_output` / `expected_answer` в структуру лога `api_responses.json`.

### 2. Поддержка локальных моделей-судей (Local Judge Support)
- **Файлы**: `eval/core/judges.py`, `eval/eval_rag_metrics.py`, `eval/config/targets.yaml`
- **Новое**: Реализован класс `LocalJudge`. Теперь система поддерживает использование любого OpenAI-совместимого сервера в качестве оценщика.
- **Особенности**: Добавлена поддержка кастомных HTTP-заголовков (необходимо для работы через туннели LocalToNet с заголовком `localtonet-skip-warning`).
- **Конфигурация**: В `targets.yaml` добавлен профиль `local-qwen`, позволяющий использовать вашу модель Qwen 35B для оценки качества.

### 3. Новый качественный датасет
- **Файл**: `eval/datasets/20260422_deep_eval_quality_dataset.json`
- **Состав**: 68 отобранных тестовых случаев по ключевым категориям (детский комплекс, SPA, горнолыжные зоны, рестораны). 
- **Чистка**: Удалены нерелевантные технические тесты (UI/Booking), оставлен фокус на качестве RAG-ответов.

### 4. Инструментарий для восстановления (Recovery Tools)
- **Файл**: `scratch/fix_api_responses.py`
- **Назначение**: Скрипт для «починки» уже существующих сломанных логов путем восстановления эталонных ответов из исходного датасета.
